# 湃乐多邮件营销系统 — 配置与启动说明

本文档说明**开发环境**、**云端 Linux 公网 IP 访问**及**生产环境**下的配置方式及启动指令。

---

## 一、开发环境（本机）

### 1. 后端配置（`backend/.env`）

在 `backend` 目录下复制 `.env.example` 为 `.env`，按需修改：

```bash
cd backend
# 若没有 .env，可复制示例（示例中为注释，需去掉 # 并填值）
copy .env.example .env   # Windows
# cp .env.example .env   # Linux / macOS
```

**开发时常用项：**

| 变量 | 说明 | 开发建议 |
|------|------|----------|
| `DATABASE_URL` | 数据库连接 | 默认 `sqlite:///./email_marketing.db` 即可 |
| `JWT_SECRET` | JWT 密钥 | 可保持默认，生产必须修改 |
| `DEBUG` | 调试模式 | 可选 `True`，便于排查问题 |
| `CORS_ORIGINS` | 允许的前端来源 | 默认 `http://localhost:5173,http://127.0.0.1:5173` |
| `AI_API_BASE_URL` / `AI_API_KEY` | AI 接口 | 不填则邮件预览为占位文案；要真实生成需填写 |
| `SMTP_*` | 发邮件 | 开发可不配，发邮件会报错 |
| `LOG_DIR` | 应用日志目录 | 默认 `logs`，日志按日期写入 `backend/logs/YYYY-MM-DD.log` |

### 2. 后端启动（开发）

```bash
cd backend
# 建议使用虚拟环境
python -m venv .venv
.venv\Scripts\activate   # Windows
conda activate email-marketing
# source .venv/bin/activate  # Linux / macOS

pip install -r requirements.txt   # 若无则安装 fastapi uvicorn 等
uvicorn app.main:app --reload --host 127.0.0.1 --port 8000
```

- 后端地址：`http://127.0.0.1:8000`
- `--reload`：代码改动自动重启（仅开发使用）

### 3. 前端配置（开发）

开发时前端通过 Vite 代理访问后端，**一般无需额外配置**：

- 请求 `/api`、`/uploads` 会被代理到 `http://127.0.0.1:8000`
- API 基地址默认 `/api`（见 `frontend/src/api/client.js`）

若后端不在本机 8000 端口，可增加环境变量（项目根下建 `frontend/.env.development`）：

```env
VITE_API_BASE_URL=http://你的后端地址:8000/api
```

### 4. 前端启动（开发）

```bash
cd frontend
npm install
npm run dev
```

- 前端地址：`http://localhost:5173`
- 浏览器访问该地址即可，接口会走代理到后端

### 5. 开发环境启动顺序小结

1. 先启动**后端**：`cd backend` → `uvicorn app.main:app --reload --host 127.0.0.1 --port 8000`
2. 再启动**前端**：`cd frontend` → `npm run dev`
3. 浏览器打开：`http://localhost:5173`

## 二、云端 Linux 服务器 + 公网 IP 访问（另一台机器浏览器访问）

代码部署在一台**云端 Linux 服务器**上，在**另一台电脑**通过公网 IP（例如 `101.132.22.43`）用浏览器访问。

### 1. 服务器上需要做的配置

#### 1.1 后端 `backend/.env`

- 后端会对外提供 API，需允许「前端页面所在来源」跨域，并建议改掉默认 JWT 密钥：

```env
# 允许来自公网 IP 的前端页面访问（端口与下面前端实际使用一致）
CORS_ORIGINS=http://101.132.22.43:5173,http://127.0.0.1:5173

# 生产建议修改
JWT_SECRET=请改为随机强密钥
DEBUG=False
```

若前端用 Nginx 挂在 80 端口，则增加：`http://101.132.22.43`（以及 `http://101.132.22.43:80` 如需要）。

#### 1.2 前端构建时的 API 地址（重要）

前端页面是在**另一台机器的浏览器**里打开的，请求会从那边发到**服务器的公网 IP**，所以构建时要填**后端公网地址**：

在 `frontend` 目录下建 `.env.production`：

```env
# 填服务器公网 IP + 后端端口（与下面启动后端用的端口一致）
VITE_API_BASE_URL=http://101.132.22.43:8000/api
```

若你有域名并做了解析，可写成：`VITE_API_BASE_URL=https://你的域名:8000/api`。

### 2. 服务器防火墙 / 安全组

在云控制台和本机防火墙中放行：

- **8000**：后端 API
- **5173** 

### 3. 在服务器上启动服务

#### 方式 A：前端用开发服务器（适合调试、临时用）

```bash
# 终端 1：后端（在项目 backend 目录下）
cd /path/to/email-marketing/backend
conda activate email-marketing  # 或 python3 -m venv .venv 后 activate
uvicorn app.main:app --host 0.0.0.0 --port 8000
```

```bash
# 终端 2：前端 dev，并允许外网访问（在项目 frontend 目录下）
cd /path/to/email-marketing/frontend
npm run dev -- --host 0.0.0.0
```

- 前端默认端口 5173，需在 `CORS_ORIGINS` 里已包含 `http://101.132.22.43:5173`。
- 另一台机器浏览器访问：**http://101.132.22.43:5173**

#### 方式 B：前端构建后本地预览服务（推荐，更接近生产）

```bash
# 1）构建前端（只需执行一次，或代码/配置改后再执行）
cd /path/to/email-marketing/frontend
# 确保已有 .env.production 且 VITE_API_BASE_URL=http://101.132.22.43:8000/api
npm run build

# 2）后端（可先在后端目录起好）
cd /path/to/email-marketing/backend
uvicorn app.main:app --host 0.0.0.0 --port 8000

# 3）在 frontend 目录另开终端：用 Vite 预览已构建产物，并对外网开放
cd /path/to/email-marketing/frontend
npx vite preview --host 0.0.0.0 --port 5173
```

- 另一台机器浏览器访问：**http://101.132.22.43:5173**
- 此时 CORS 需包含 `http://101.132.22.43:4173`（上面已在示例中写出）。

### 4. 在另一台机器上访问

- 浏览器地址栏输入：
  - 方式 A：`http://101.132.22.43:5173`
  - 方式 B：`http://101.132.22.43:4173`
- 若无法打开：先确认服务器上后端、前端进程在跑，再检查云安全组和本机防火墙是否放行对应端口。

### 5. 小结（公网 IP 101.132.22.43 示例）

| 步骤 | 说明 |
|------|------|
| 后端 .env | `CORS_ORIGINS` 包含 `http://101.132.22.43:5173` 
| 前端 .env.production | `VITE_API_BASE_URL=http://101.132.22.43:8000/api` |
| 防火墙/安全组 | 放行 8000、5173 或 4173 |
| 后端启动 | `uvicorn app.main:app --host 0.0.0.0 --port 8000` |
| 前端 | 方式 A：`npm run dev -- --host 0.0.0.0` → 访问 **:5173**；方式 B：`npm run build` + `npx vite preview --host 0.0.0.0 --port 4173` → 访问 **:4173** |

## 三、生产环境（域名 + Nginx 等）

### 1. 后端配置（`backend/.env`）

生产环境**必须**单独维护一份 `.env`（不要与开发混用），并至少修改以下项：

| 变量 | 说明 | 生产要求 |
|------|------|----------|
| `JWT_SECRET` | JWT 签名密钥 | **必须**改为随机强密钥 |
| `DEBUG` | 调试模式 | **必须**为 `False` |
| `DATABASE_URL` | 数据库 | 若用 MySQL/PostgreSQL，改为对应连接串 |
| `CORS_ORIGINS` | 允许的前端来源 | 填生产前端域名，如 `https://your-domain.com` |
| `SMTP_HOST` / `SMTP_USER` / `SMTP_PASSWORD` / `SMTP_SENDER` | 发件 | 填真实 SMTP，否则无法发邮件 |
| `AI_API_BASE_URL` / `AI_API_KEY` | AI 接口 | 按需填写，否则邮件预览为占位 |
| `UPLOAD_DIR` | 上传目录 | 可按部署路径改为绝对路径 |

### 2. 后端启动（生产）

**不**使用 `--reload`，并建议绑定到 `0.0.0.0` 以便对外访问：

```bash
cd backend
# 使用生产用虚拟环境
uvicorn app.main:app --host 0.0.0.0 --port 8000
```

- 多进程示例（按需使用，注意与定时任务兼容）：
  ```bash
  uvicorn app.main:app --host 0.0.0.0 --port 8000 --workers 2
  ```
- 若通过 Nginx 等反向代理，可只监听本机：`--host 127.0.0.1 --port 8000`，由 Nginx 转发到 80/443。

### 3. 前端配置（生产）

生产环境前端通常**单独构建**并部署到静态服务器或 CDN，需要让请求发到**真实后端地址**：

在 `frontend` 目录下建 `.env.production`（或构建时传入环境变量）：

```env
VITE_API_BASE_URL=https://你的后端域名/api
```

例如后端为 `https://api.example.com`，则填：

```env
VITE_API_BASE_URL=https://api.example.com/api
```

### 4. 前端构建与预览（生产）

```bash
cd frontend
npm install
npm run build
```

- 产物在 `frontend/dist`，将该目录部署到 Nginx、Caddy 或任意静态托管即可。
- 本地验证构建结果：
  ```bash
  npm run preview
  ```
  会启动静态服务（默认如 `http://localhost:4173`），此时需保证 `VITE_API_BASE_URL` 指向可访问的后端。

### 5. 生产环境部署顺序小结

1. **后端**：在服务器上配置好 `backend/.env`，执行  
   `uvicorn app.main:app --host 0.0.0.0 --port 8000`
2. **前端**：配置 `frontend/.env.production` 中的 `VITE_API_BASE_URL`，执行  
   `npm run build`，将 `frontend/dist` 部署到 Web 服务器
3. **反向代理**（推荐）：用 Nginx 等将前端域名指向 `dist`，将 `/api`、`/uploads` 反向代理到后端 `http://127.0.0.1:8000`，这样前端可继续用相对路径 `/api`，无需改 `VITE_API_BASE_URL`

## 四、Linux 生产部署：后台运行 + 断线自启（推荐方案）

适用于将服务部署到 Linux 服务器，要求**后台常驻**、**崩溃自启**、**开机自启**，并与其他服务（8765、80、443）共存。

### 1. 端口规划（与其他服务共存）

| 端口 | 用途 |
|------|------|
| **8000** | 本系统后端 API |
| **5173** | 本系统前端（或由 80/443 的 Nginx 提供） |
| **8765** | 你的其他服务 |
| **80 / 443** | 你的其他服务（或 Nginx 统一入口） |

云防火墙已放行 8000、5173 即可。

### 2. 方案 A：systemd 独立运行（前后端分离，8000 + 5173）

前后端分别用 systemd 管理，各自监听 8000 和 5173，**崩溃自动重启**，**开机自动启动**。

#### 2.1 项目部署目录

假设项目在 `/opt/email-marketing`（可按实际路径修改）：

```bash
# 拉取/上传代码后
cd /opt/email-marketing/backend
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

cd /opt/email-marketing/frontend
npm install
npm run build
```

#### 2.2 后端 systemd 服务

创建 `/etc/systemd/system/email-marketing-api.service`：

```ini
[Unit]
Description=Email Marketing API (FastAPI)
After=network.target

[Service]
Type=simple
User=www-data
Group=www-data
WorkingDirectory=/opt/email-marketing/backend
Environment="PATH=/opt/email-marketing/backend/.venv/bin"
ExecStart=/opt/email-marketing/backend/.venv/bin/uvicorn app.main:app --host 0.0.0.0 --port 8000
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

- `Restart=always`：进程退出后自动重启（含崩溃）
- `User` / `Group`：按你实际运行用户改（如 `root` 或 `deploy`）

#### 2.3 前端 systemd 服务（静态文件）

前端用 **构建后的静态文件**，用 `serve` 或 `npx serve` 提供 HTTP 服务，**不要用** `npm run dev`。

```bash
# 安装 serve（一次性）
npm install -g serve
```

创建 `/etc/systemd/system/email-marketing-web.service`：

```ini
[Unit]
Description=Email Marketing Web (Frontend)
After=network.target

[Service]
Type=simple
User=www-data
Group=www-data
WorkingDirectory=/opt/email-marketing/frontend
ExecStart=/usr/bin/npx serve -s dist -l 5173
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

若 `npx serve` 路径不同，可用 `which serve` 或 `which npx` 确认后替换 `ExecStart`。

#### 2.4 前端构建配置

在 `frontend/.env.production` 中配置后端地址（公网 IP 或域名）：

```env
VITE_API_BASE_URL=http://你的公网IP:8000/api
```

每次改完前端代码或该配置后，需重新构建：

```bash
cd /opt/email-marketing/frontend
npm run build
```

#### 2.5 启用并管理服务

```bash
sudo systemctl daemon-reload
sudo systemctl enable email-marketing-api email-marketing-web
sudo systemctl start email-marketing-api email-marketing-web
sudo systemctl status email-marketing-api email-marketing-web
```

常用命令：
- `sudo systemctl restart email-marketing-api`：重启后端
- `sudo systemctl restart email-marketing-web`：重启前端
- `journalctl -u email-marketing-api -f`：查看后端日志

### 3. 方案 B：Nginx 统一入口（80/443，推荐有域名时使用）

若 80/443 由你掌控，可用 Nginx 做反向代理：  
- 前端：`/` 或子路径  
- 后端：`/api`、`/uploads` 代理到 `127.0.0.1:8000`  

这样只需开放 80/443，**不暴露** 8000、5173 到公网，更安全。

生产环境（方案 B：Nginx）
生产部署时，前端不要用 npm run dev，而是：
用 Nginx 提供前端：直接托管 dist 目录
后端用 systemd 管理：例如用 conda 环境运行 uvicorn

整体顺序可以这样理解：
1. 部署代码到服务器（git clone 或 上传）
2. 安装依赖：cd frontend && npm install
3. 配置 .env.production（VITE_API_BASE_URL=/api）
4. 执行 npm run build 更新 dist 目录
5. 配置 Nginx（指向 dist 目录）
6. 配置并启动 systemd 后端
7. 重载 Nginx

#### 第一步：前端

构建前端：npm run build → 生成 dist

1. 新建配置文件
sudo nano /etc/nginx/sites-available/email-marketing
写入下面内容（注意把路径和域名改成你的实际值）：

# 可选：80 自动跳转到 443
server {
    listen 80;
    server_name mail.pltplt.com;
    return 301 https://$host$request_uri;
}
# 443 HTTPS
server {
    listen 443 ssl;
    server_name mail.pltplt.com;
    ssl_certificate     /etc/nginx/ssl/_.pltplt.com/_.pltplt.com.pem;
    ssl_certificate_key /etc/nginx/ssl/_.pltplt.com/_.pltplt.com.key;
    gzip on;
    gzip_min_length 1k;
    gzip_comp_level 6;
    gzip_types text/plain application/javascript application/x-javascript text/css application/xml text/javascript application/json image/svg+xml;
    gzip_vary on;
    root /data/plt-email-marketing/frontend/dist;
    index index.html;
    location / {
        try_files $uri $uri/ /index.html;
        expires 1h;
    }
    location /api {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
    location /uploads {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
    }
}
2. 启用并重载

# 创建软链接启用站点
sudo ln -s /etc/nginx/sites-available/email-marketing /etc/nginx/sites-enabled/
# 测试配置
sudo nginx -t
# 重载生效
sudo systemctl reload nginx

#### 第二步：后端

创建 systemd 服务文件
sudo nano /etc/systemd/system/email-marketing-api.service
写入：

[Unit]
Description=Email Marketing API (FastAPI)
After=network.target
[Service]
Type=simple
User=root
Group=root
WorkingDirectory=/data/plt-email-marketing/backend
ExecStart=/root/miniconda3/envs/email-marketing/bin/uvicorn app.main:app --host 127.0.0.1 --port 8000
Restart=always
RestartSec=5
[Install]
WantedBy=multi-user.target


启用并启动服务

# 加载新的 unit 文件
sudo systemctl daemon-reload
# 开机自启
sudo systemctl enable email-marketing-api
# 立即启动
sudo systemctl start email-marketing-api
# 查看状态
sudo systemctl status email-marketing-api

**第三步：常用管理命令**

| 命令 | 说明 |
|------|------|
| `sudo systemctl start email-marketing-api` | 启动 |
| `sudo systemctl stop email-marketing-api` | 停止 |
| `sudo systemctl restart email-marketing-api` | 重启（如更新代码后） |
| `sudo systemctl status email-marketing-api` | 查看状态 |
| `journalctl -u email-marketing-api -f` | 实时查看日志 |
| `journalctl -u email-marketing-api -n 100` | 最近 100 行日志 |

**前端更新流程**：改完前端代码后，在 `frontend` 目录执行 `npm run build`，Nginx 会自动读取新的 `dist` 目录，无需重启服务。

### 4. 方案选择建议

| 场景 | 建议 |
|------|------|
| 无域名，仅用公网 IP | 方案 A（8000 + 5173，systemd 双服务） |
| 有域名，80/443 可自用 | 方案 B（Nginx + 仅后端 systemd） |
| 80/443 已被其他站点占用 | 方案 A，或用子路径/子域名接入 Nginx |

### 5. 与其他服务（8765、80、443）共存

- **8765**：与本系统的 8000、5173 互不冲突，直接并存。
- **80/443**：若已有其他站点，可在同一 Nginx 中新增一个 `server` 块，用不同 `server_name` 或 `location` 区分本系统与其他服务。
